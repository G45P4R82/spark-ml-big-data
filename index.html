O Desafio do Big Data

A Explosão de Dados

  • Geração sem precedentes por Redes Sociais, IoT, E-commerce, etc.
  • Os 3 V's do Big Data limitam as ferramentas analíticas convencionais:
    • Volume: Terabytes e Petabytes.
    • Velocidade: Fluxo contínuo (streaming).
    • Variedade: Dados estruturados e não-estruturados.

O Limite Tradicional

  • Bancos de dados relacionais em um único nó não escalam fisicamente (Vertical Scaling).
  • Técnicas analíticas antigas não encontram padrões ocultos complexos em grandes massas de dados.
  • Solução: Computação Distribuída (Escalonamento Horizontal) + Machine Learning.
IC - Unicamp | Juan Nascimento Slide 2 / 17

O Paradigma Hadoop & MapReduce

A Primeira Geração de Processamento Distribuído

  • Como Funciona: Divide o trabalho em milhares de máquinas. O Map filtra/ordena, o Reduce sumariza os resultados.
  • O Gargalo Crítico (I/O): É um framework focado em Disco (HDFS).
    • Entre cada etapa de Map e Reduce, os dados intermediários são gravados fisicamente nos discos rígidos do cluster.
Problema para Machine Learning:
Algoritmos de ML são fortemente iterativos (repetem cálculos sobre o mesmo dado milhares de vezes, ex: Gradient Descent). O I/O constante de disco do Hadoop torna o treinamento inviável ou absurdamente lento.
IC - Unicamp | Juan Nascimento Slide 3 / 17

A Revolução do Apache Spark

Processamento In-Memory

  • Diferente do Hadoop, o Spark processa os dados inteiramente na Memória RAM (In-Memory Computing).
  • Evita I/O de disco: Dados intermediários ficam cacheados na memória entre as iterações.
  • Pode ser de 10x a 100x mais rápido que o Hadoop MapReduce para cargas de trabalho iterativas.

Ideal para as operações intensivas de otimização matemática exigidas pelos modelos de Machine Learning (como backpropagation e divisão de árvores).

Mantém tolerância a falhas (via RDDs/DataFrames) mesmo operando em memória volátil.

O que é o Spark MLlib?

A biblioteca nativa e unificada de Machine Learning do Apache Spark.

  • Algoritmos Distribuídos: Implementações de Regressão, Classificação, Clustering e Filtragem Colaborativa projetadas desde o zero para rodar em clusters.
  • Pipeline API: Inspirada no Scikit-Learn. Permite encadear transformações de dados e treinamento de modelos de forma modular.
  • Tuning Simplificado: Suporte nativo a Grid Search e Cross-Validation rodando paralelamente em vários nodos (Workers).
  • Operações Matriciais: Algebra linear escalável (BLAS local + distribuição inteligente) otimizada para DataFrames de alta dimensionalidade.

Workflow de ML no Spark

Toda essa arquitetura é executada em paralelo através de dezenas de "Executors" sobre o gerenciador de recursos YARN, processando enormes blocos de dados contíguos simultaneamente.

Motivação do Artigo

O Objetivo Principal

Enquanto muitos estudos focam no desenvolvimento teórico de novos algoritmos de Machine Learning, existe um gap na aplicação prática (deployment) de modelos clássicos em cenários industriais reais de Big Data.

A pesquisa busca responder:

  • Quão escaláveis são as implementações distribuídas nativas do Apache Spark?
  • Qual é o verdadeiro Trade-off entre Performance Preditiva (Acurácia/RMSE) e Custo Computacional (Tempo, CPU e RAM) em datasets da ordem de 100 milhões de registros?

Modelos Baseados em Árvores

Random Forest (RF)

  • Metodologia: Cria múltiplas árvores de decisão de forma independente (via bootstrap/bagging) e tira a média (ou votação) no final.
  • No Spark: Facilmente paralelizável. Cada worker pode treinar um subconjunto de árvores quase sem comunicação de rede.
  • Vantagem: Excelente prevenção de overfitting.

Gradient-Boosted Trees (GBT)

  • Metodologia: Iterativo e sequencial. Cada nova árvore é treinada para corrigir os erros residuais combinados de todas as árvores anteriores.
  • No Spark: Desafiador. Cada iteração obriga os nós a se comunicarem (network shuffle) para atualizar os resíduos globais.
  • Vantagem: Altíssima precisão.
IC - Unicamp | Juan Nascimento Slide 10 / 17

Modelos Lineares e de Instância

Regressão Linear (LinR) e Logística (LR):

Calculam equações e hiperplanos simples. São extremente rápidos e escalam perfeitamente na arquitetura Spark. Excelente "baseline" ou para cenários de tempo real (latência baixíssima).

Support Vector Machines (SVM / SVR):

Tentam maximizar a margem entre classes ou erros. Com kernel linear no Spark, oferecem um bom meio-termo entre recursos exigidos e precisão preditiva, mas sofrem computacionalmente com o aumento do volume de dados.

K-Nearest Neighbors (KNN) - O Vilão:

Como não há "treinamento" real (apenas armazenamento), a fase de predição é trágica em ambientes distribuídos. Para inferir um ponto, ele calcula a distância contra os 100 milhões de registros existentes na memória, causando estouro de RAM e extrema lentidão de inferência.

Insights Finais & Conclusões

  • A Viabilidade do Spark: Todos os modelos avaliados demonstraram crescimento computacional perfeitamente linear à medida que o tamanho dos dados escalou, validando a arquitetura in-memory.
  • Trade-off Primário:
    • Aplicações que demandam Baixa Latência / Tempo Real / Baixo Orçamento: Optar por Linear / Logistic Regression.
    • Aplicações de Missão Crítica (Ciência, Detecção Severa) com Infraestrutura farta: O Gradient-Boosted Trees (GBT) justifica seu alto custo computacional entregando precisões superiores a AUC 0.97.
  • Cuidado Arquitetural: Algoritmos dependentes de instâncias locais ou de distância euclidiana exata contra todo o banco (como KNN) enfrentam sérios gargalos de rede (Shuffle) e não são indicados para o Spark MLlib nativo em escalas de 100M+ dados.
